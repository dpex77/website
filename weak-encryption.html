1. Overview

Cryptographic failures occur when sensitive data is protected using weak, outdated, or improperly implemented cryptography. Such failures can expose data to interception, tampering, or unauthorized access.

2. Common Causes

2.1 Broken or Deprecated Algorithms - Symmetric/Asymmetric: DES, 3DES, RC4, MD5, SHA-1 - Message Authentication Codes: Insecure HMAC implementations (e.g., HMAC-MD5) - Weak key exchange: Small Diffie-Hellman groups (512/768-bit), export-grade ciphers 2.2 Outdated TLS Versions - TLS 1.0 and 1.1 vulnerable to attacks like POODLE, BEAST - Recommended: TLS 1.2+ (TLS 1.3 preferred) 2.3 Weak Keys - Short RSA keys (<2048 bits) → brute-force feasible - Weak ECC curves (P-192) → insufficient security margin - Static or reused keys → predictable and exploitable 2.4 Insecure Key Storage - Keys stored in plaintext on disk or in source code - Keys in environment variables or unprotected config files - Lack of HSM or secure vaults (HashiCorp Vault, AWS KMS) 2.5 Weak Hashing Algorithms - MD5, SHA-1 (collision-prone) - Unsalted password hashes → rainbow table attacks - Inadequate key derivation (avoid fast hashes; prefer Argon2, bcrypt, PBKDF2)

3. Attacker Perspective

- Exploit weak or broken algorithms to decrypt sensitive data - Perform collision attacks on MD5/SHA-1 to forge data or signatures - Steal keys from improperly secured storage - Intercept communications over outdated TLS - Brute-force short or reused keys

4. Defender Perspective

- Using deprecated ciphers or TLS versions increases exposure - Weak keys and poor storage make key compromise likely - Insecure password hashing allows mass password cracking if breached - Lack of key rotation increases the impact of a compromised key

5. Real-World Examples

1. POODLE Attack (2014) - Exploited SSL 3.0 fallback - Allowed attackers to decrypt secure cookies over HTTPS 2. MD5 Collision Attacks - Certificates and digital signatures forged by exploiting MD5 collisions 3. Heartbleed Vulnerability - While a memory bug, it exposed private keys stored in server memory 4. SolarWinds SUNBURST - Integrity failure combined with improper cryptography allowed signed malicious updates

6. Best Practices / Mitigation

6.1 Use Strong, Modern Cryptography - TLS 1.2/1.3 only, strong cipher suites - Minimum key sizes: RSA ≥ 2048 bits, ECC ≥ 256 bits (P-256, Curve25519) 6.2 Secure Key Storage - Hardware Security Modules (HSM) or Key Vaults - Avoid plaintext or unprotected storage - Separate storage for salts/peppers in password hashing 6.3 Strong Password Hashing - Argon2, bcrypt, PBKDF2 - Use unique salt for each password - Optionally add a pepper for additional security 6.4 Key Management - Rotate keys regularly - Revoke compromised keys immediately - Use ephemeral session keys where possible 6.5 Continuous Verification - Scan servers for weak ciphers (SSLyze, testssl.sh) - Monitor certificates, check for expired or weak keys - Audit cryptography usage in applications

7. Summary

Cryptographic failures can compromise confidentiality, integrity, and authentication of sensitive data. Using strong, modern algorithms, secure key storage, proper hashing, and regular auditing are essential to mitigate these risks.
