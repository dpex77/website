1. Overview:

XSS (Cross-Site Scripting) is a vulnerability that allows attackers to inject malicious scripts into web pages viewed by others. Once executed in a victim’s browser, these scripts can: - Steal cookies or session tokens - Redirect users to malicious sites - Install malware - Perform phishing attacks

2. Vulnerability from an Attacker Perspective:

XSS occurs when user input is not properly sanitized or encoded, allowing scripts such as: <!-- <script>malicious_code</script> --> to run directly in the browser.

2.1 Types of XSS / Attempts of an Attacker:

Reflected XSS: - Payload embedded in a crafted link or request. - Server reflects the payload in response without storing it. - Requires user interaction: attacker tricks victim into clicking link. Stored XSS: - Payload saved on the server (comment boxes, profile fields, forum posts, chat). - Example: <!-- <script>alert('Hacked!')</script> --> - No attacker interaction needed after payload stored. DOM-Based XSS: - Vulnerability exists entirely in client-side JavaScript. - Example: document.getElementById('output').textContent = location.hash; - Unsafe example commented: <!-- <script>alert('XSS')</script> --> - Triggered purely by unsafe DOM manipulation.

3. Vulnerability from a Defender Perspective:

- Input is not validated or sanitized. - User-provided content is inserted into page without encoding. - Scripts execute in other users’ browsers, potentially stealing data or performing actions.

3.1 Solutions for a Defender:

- Input Validation: Reject or filter dangerous inputs (<, >, <script>). - Output Encoding: Encode user data before rendering in HTML, JS, or URLs. - Content Security Policy (CSP): Restrict which scripts can execute. - HTTP-only Cookies: Prevent JavaScript from accessing session data. - Safe DOM Handling: Avoid inserting untrusted data into innerHTML; use textContent or safe templating libraries.

4. Extras:

- XSS attacks rely on the victim executing attacker-supplied scripts. - Stored XSS is more dangerous than reflected XSS due to persistence. - DOM-based XSS can occur without server-side vulnerabilities, purely via unsafe client-side code.
