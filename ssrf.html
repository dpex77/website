1. Overview:

Server-Side Request Forgery (SSRF) is a vulnerability where an attacker tricks a server into making requests on their behalf. This can target internal systems or external services that the attacker cannot access directly.

2. Vulnerability from an Attacker Perspective:

The web application accepts user-supplied URLs or resources (file fetch, image download, webhook, API call) and the server performs the request without proper validation.

2.1 Attempts of an Attacker:

Attack Targets: - Internal Services: http://localhost:8080, internal APIs, Redis, Jenkins, etc. - Private IP Ranges: 10.x.x.x, 192.168.x.x, 127.0.0.1 - Cloud Metadata Services: AWS http://169.254.169.254, GCP, Azure Why SSRF is Dangerous: - Bypasses Firewalls – Internal services often trust requests from internal servers. - Accesses Sensitive Internal Services – May reveal admin panels or internal APIs. - Cloud Metadata Theft – Extracts credentials (IAM keys, tokens) from cloud services. - Pivoting / Lateral Movement – Can scan internal networks or reach otherwise inaccessible systems.

3. Vulnerability from a Defender Perspective:

- Server accepts user input and performs server-side HTTP requests without validation. - Can expose internal services and sensitive endpoints. - May lead to credential theft or network compromise if exploited.

3.1 Solutions for a Defender:

- Whitelist allowed domains/IPs and reject all others. - Never directly use user input in server-side requests. - Block internal IP ranges (127.0.0.1, 10.0.0.0/8, etc.). - Network segmentation – Limit backend services’ access to internal networks. - Monitor and log outgoing requests for suspicious patterns.

4. Extras:

- SSRF can combine with other vulnerabilities (RCE, data exfiltration). - Careful input validation and strict network policies are essential.
