1. Overview / Scenario:

An authenticated user is browsing a site and has a valid authentication cookie/session. An attacker tricks the user into performing an action (e.g., clicking a malicious link, visiting a crafted page via phishing or XSS). This redirects or triggers requests to a legitimate site where the user is authenticated. The attacker doesn’t need the user’s password. CSRF exploits the fact that the server trusts requests from the authenticated client.

2. Why it works:

Active session: The user is already logged in; the browser sends the authentication cookie automatically. Same-Origin Policy (SOP): Browsers restrict scripts from one origin from reading data on another origin. Limitation: SOP does not prevent the browser from sending cookies with legitimate requests to the target domain.

3. Mitigations:

1. Cookie-based mitigations: SameSite Attribute: Controls when cookies are sent in cross-site requests: - SameSite=Strict: Cookies are sent only for requests from the same origin. - SameSite=Lax: Cookies are sent for top-level navigations but not for embedded requests (like images or iframes). - SameSite=None: Cookies are sent with all requests but must be marked Secure (HTTPS only). Purpose: Limits automatic cookie sending in cross-origin requests.

4. CSRF Tokens:

Definition: Unique, unpredictable token issued per user session (or per request). How it works: 1. Server generates a CSRF token and associates it with the user’s session. 2. Token is sent to the client via: - Hidden HTML form fields - HTTP headers for AJAX requests (e.g., X-CSRF-Token) 3. When the client submits a request, the server verifies that the token matches the session-stored value. Key points: - Tokens are unique and unpredictable. - Tokens expire after some time. - Tokens ensure that only legitimate requests from the authenticated user are accepted.
