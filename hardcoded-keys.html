1. Overview

Hardcoded keys and secrets refer to credentials, API keys, cryptographic keys, tokens, or passwords embedded directly in source code, binaries, or configuration files. These can be extracted by attackers, compromising the security of applications, services, or sensitive data.

2. Common Causes

2.1 Embedded Secrets in Source Code - API keys, passwords, encryption keys written directly in code. - Often committed to version control (GitHub, GitLab, etc.). 2.2 Configuration Files - Secrets stored in plaintext config files shipped with applications. 2.3 Compiled Binaries / Mobile Apps - Keys embedded in executables, libraries, or mobile apps. - Reverse engineering can reveal them. 2.4 Third-Party Libraries / SDKs - Libraries containing their own hardcoded keys or tokens. - Developers may inadvertently include them.

3. Attacker Perspective

- Extract keys from public repositories, code, or binaries. - Access sensitive resources without authorization (databases, APIs, cloud services). - Decrypt data encrypted with hardcoded secrets. - Use the same key across environments to escalate privileges.

4. Defender Perspective

- Hardcoded secrets are a single point of failure. - They are difficult to rotate without redeploying applications. - Exposure may result in widespread compromise. - Violates compliance requirements for secure secret management.

5. Examples

Example 1: API Key in Source Code const API_KEY = "12345abcdef"; - Anyone inspecting the code can access the service with full privileges. Example 2: Database Credentials in Configuration db_user = "admin" db_pass = "P@ssw0rd123" - If leaked, attacker can connect directly to the database. Example 3: Hardcoded Encryption Key const ENCRYPTION_KEY = "abcdef1234567890"; - All encrypted user data can be decrypted if the key is discovered. Example 4: Embedded OAuth Token const ACCESS_TOKEN = "ya29.A0ARrdaM...xyz"; - Enables full API access without user consent or authentication.

6. Best Practices / Mitigation

6.1 Avoid Hardcoding Secrets - Never store keys, passwords, or tokens directly in code. 6.2 Use Secure Secret Management - Environment variables for runtime injection. - Dedicated secret management systems (HashiCorp Vault, AWS KMS, Azure Key Vault). - Encrypted configuration files with restricted access. 6.3 Rotate Keys Regularly - Replace keys periodically or immediately if compromised. 6.4 Limit Scope and Permissions - Apply least privilege principle to all keys and secrets. - Avoid using same keys across environments or services. 6.5 Monitor for Leaks - Scan source code and repositories for secrets (GitLeaks, TruffleHog). - Audit binaries, containers, and mobile apps for embedded secrets.

7. Summary

Hardcoded keys and secrets are critical security vulnerabilities. They can be easily extracted, giving attackers unauthorized access to sensitive resources. Using secure secret management, proper access controls, key rotation, and regular audits are essential to mitigate this risk.
